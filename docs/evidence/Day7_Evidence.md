## 6. AI 协作复盘

**AI 在哪些阶段帮助最大：**
AI 在 Day2（页面骨架和路由配置）、Day3（Mock 数据填充和 API 封装）、Day6（交互组件生成）三个阶段帮助最为明显，这些任务具有模板化和重复性高的特点，AI 能快速生成初稿，人工只需调整细节即可复用。

**AI 生成内容中出现过哪些问题：**
最常见的问题是 AI 生成的功能超出了项目范围。例如在 Day4 发布表单中，AI 自动加入了图片上传组件和文件大小校验；在 Day5 用户相关代码中，AI 试图引入 JWT 登录和注册页面。此外，AI 生成的 Mock 数据偏通用电商风格，字段名称不够贴合校园业务场景。还有组件抽象过度的问题——AI 建议为仅在一个页面使用的功能抽取独立组件，增加了不必要的维护成本。

**自己如何判断和修改 AI 生成内容：**
判断原则是"严格对照每日任务清单"，超出当天任务的 AI 建议一律不采用或留到后续阶段。对于数据字段，逐条检查语义是否适合校园场景，例如将"商品描述"改为"物品描述"。代码风格方面，统一调整 AI 生成的代码风格以匹配项目已有规范。类型安全方面，每次 AI 生成接口代码后都用 vue-tsc 验证。

**哪些内容必须由自己理解后才能完成：**
项目的业务逻辑理解（四个业务类型的差异和字段设计）、功能范围控制（什么该做、什么不该做）、状态边界划分（如何拆分 Store）、路由组织结构、组件命名和目录层级等架构层面的决策，都必须由自己理解后才能完成。

**以后使用 AI Coding 工具时会注意什么：**
会更加注意审查 AI 生成的内容是否在项目范围内，不被 AI 带偏方向；会对 AI 生成的数据字段进行更细致的业务贴合检查；会提前明确功能边界，减少 AI 生成的无效代码；会在 AI 生成后立即运行类型检查和构建，及早发现问题。

7. **问题与解决**（20s）— 表单字段切换不清除校验、JSON Server 重启后 ID 变化等问题及处理

## 8. 项目不足与改进方向

- 目前使用 JSON Server 模拟后端，数据在内存中，重启后新增数据丢失，后续可对接真实后端接口
- 用户状态通过 Pinia 模拟，无真实登录注册流程，后续可加入 JWT 认证
- 收藏数据存储在 Pinia 内存中，页面刷新后丢失，后续可持久化到 localStorage 或后端
- 未实现图片上传功能，发布表单中图片字段为文本输入，后续可集成文件上传组件
- 搜索功能仅前端关键字过滤，数据量大时效率低，后续可改为后端分页搜索
- 页面样式和响应式适配还有优化空间，当前主要适配 PC 端
- 缺少详情页：当前点击卡片跳转的是其他页面，未实现真正的商品详情页

## 9. 今日反思

通过 7 天的实训，我深刻理解了一个前端项目从零到一的完整过程：从需求分析和页面清单（Day1），到页面骨架和路由搭建（Day2），再到数据层和列表渲染（Day3），然后是交互层如表单和状态管理（Day4/5），最后是体验优化和验收交付（Day6/7）。每一步都有其不可跳过的价值。

让我收获最大的是对"工程化"的理解——项目不只是写代码，还包括目录结构设计、组件拆分粒度、状态边界划分、 Git 提交规范、文档记录和证据留存。特别是在 AI 协作方面，我学会了如何让 AI 做它擅长的事（生成模板代码、填充数据），同时保持自己对架构和业务的掌控。AI 可以提高效率，但不能替代对业务的理解和代码质量的判断。

这次实训让我意识到，真正的开发能力不是把功能做出来，而是知道什么时候该用什么技术、如何组织代码、如何取舍功能、如何让项目可持续维护。Day7 的综合验收不是结束，而是将从实训中学到的工程化思维应用到未来项目的开始。
